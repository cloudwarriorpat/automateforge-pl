# Checklist: Spotkanie kickoff z klientem

**Autor:** AutomateForge.pl
**Wersja:** 1.0
**Ostatnia aktualizacja:** 2025-01
**Czas trwania spotkania:** 60-90 minut

---

## Cel spotkania

Spotkanie kickoff jest pierwszym formalnym spotkaniem projektowym z klientem po podpisaniu
umowy. Jego celem jest potwierdzenie zakresu, ustalenie harmonogramu, zebranie wymaganych
informacji i zbudowanie relacji z zespołem klienta.

---

## Przed spotkaniem (przygotowanie)

### Dokumenty do przygotowania
- [ ] Raport diagnostyczny (wygenerowany wcześniej)
- [ ] Podpisana umowa (kopia dla referencji)
- [ ] Prezentacja kickoff (szablon w ops/templates/)
- [ ] Formularz zbierania dostępów
- [ ] Wstępny harmonogram projektu

### Logistyka
- [ ] Potwierdzenie terminu z klientem (min. 2 dni wcześniej)
- [ ] Wysłanie zaproszenia z agendą
- [ ] Przygotowanie linku do wideokonferencji (lub rezerwacja sali)
- [ ] Test połączenia (jeśli online)
- [ ] Przygotowanie narzędzia do notatek

### Uczestnicy spotkania

**Strona AutomateForge.pl:**
- [ ] Opiekun klienta (prowadzący spotkanie)
- [ ] Architekt rozwiązania / Lead developer
- [ ] (Opcjonalnie) Specjalista od KSeF / AI

**Strona klienta (poproś o obecność):**
- [ ] Osoba decyzyjna / sponsor projektu
- [ ] Osoba odpowiedzialna za projekt
- [ ] Osoba techniczna / administrator IT
- [ ] (Opcjonalnie) Przedstawiciel księgowości

---

## Agenda spotkania

### 1. Powitanie i przedstawienie zespołów (10 min)

- [ ] Powitanie uczestników
- [ ] Przedstawienie zespołu AutomateForge.pl (role, odpowiedzialności)
- [ ] Przedstawienie zespołu klienta
- [ ] Krótkie omówienie agendy spotkania

### 2. Potwierdzenie informacji o kliencie (10 min)

- [ ] Weryfikacja danych firmy:
  - [ ] Pełna nazwa firmy
  - [ ] NIP
  - [ ] Adres siedziby
  - [ ] Branża
- [ ] Potwierdzenie osób kontaktowych:
  - [ ] Sponsor projektu: _______________
  - [ ] Project Manager klienta: _______________
  - [ ] Kontakt techniczny: _______________
  - [ ] Kontakt księgowość: _______________
- [ ] Potwierdzenie danych z raportu diagnostycznego:
  - [ ] Wykorzystywane systemy IT
  - [ ] Wolumen faktur
  - [ ] Zidentyfikowane problemy

### 3. Przegląd zakresu projektu (15 min)

- [ ] Omówienie zakresu z umowy
- [ ] Potwierdzenie priorytetów wdrożenia:
  - [ ] Priorytet 1: _______________
  - [ ] Priorytet 2: _______________
  - [ ] Priorytet 3: _______________
- [ ] Omówienie co jest w zakresie, a co poza zakresem
- [ ] Potwierdzenie wybranego pakietu (Starter / Professional / Enterprise)
- [ ] Dyskusja o oczekiwanych rezultatach
- [ ] Ustalenie kryteriów sukcesu (mierzalne KPI):
  - [ ] KPI 1: _______________
  - [ ] KPI 2: _______________
  - [ ] KPI 3: _______________

### 4. Wymagania dostępowe (15 min)

- [ ] Omówienie potrzebnych dostępów i danych
- [ ] System ERP / księgowy:
  - [ ] Nazwa i wersja systemu: _______________
  - [ ] Typ dostępu (API / panel / VPN): _______________
  - [ ] Kto nadaje dostęp? _______________
  - [ ] Termin uzyskania dostępu: _______________
- [ ] KSeF:
  - [ ] Czy firma ma konto w KSeF? TAK / NIE
  - [ ] Typ autoryzacji (token / certyfikat): _______________
  - [ ] Kto zarządza certyfikatami? _______________
- [ ] Email (jeśli agent email):
  - [ ] Adres email do monitorowania: _______________
  - [ ] Kto wygeneruje hasło aplikacji? _______________
- [ ] Slack / Teams (komunikacja):
  - [ ] Który kanał preferowany? _______________
  - [ ] Kto utworzy kanał / doda nas? _______________
- [ ] Dane testowe:
  - [ ] Przykładowe faktury (min. 5-10 szt.)
  - [ ] Przykładowe emaile (jeśli agent email, min. 20-30 szt.)
  - [ ] Termin dostarczenia: _______________

### 5. Harmonogram i kamienie milowe (15 min)

- [ ] Prezentacja wstępnego harmonogramu
- [ ] Ustalenie dat kluczowych kamieni milowych:

| Kamień milowy | Planowana data | Status |
|---------------|---------------|--------|
| Kickoff (dzisiaj) | [DATA] | Zrealizowany |
| Uzyskanie wszystkich dostępów | [DATA] | Oczekujący |
| Zakończenie audytu technicznego | [DATA] | Oczekujący |
| Gotowość środowiska testowego | [DATA] | Oczekujący |
| Zakończenie testów | [DATA] | Oczekujący |
| Go-live | [DATA] | Oczekujący |
| Zakończenie monitoringu pow. | [DATA] | Oczekujący |

- [ ] Ustalenie częstotliwości spotkań statusowych:
  - [ ] Tygodniowo
  - [ ] Co dwa tygodnie
  - [ ] Inny: _______________
- [ ] Ustalenie dnia i godziny spotkań statusowych: _______________
- [ ] Identyfikacja znanych ryzyk i ograniczeń czasowych (urlopy, zamknięcie miesiąca itp.)

### 6. Plan komunikacji (10 min)

- [ ] Potwierdzenie kanałów komunikacji:
  - [ ] Email: _______________
  - [ ] Slack/Teams: _______________
  - [ ] Telefon (pilne): _______________
- [ ] Ustalenie zasad komunikacji:
  - [ ] Czas reakcji na email: _______________
  - [ ] Czas reakcji na Slack: _______________
  - [ ] Procedura eskalacji: _______________
- [ ] Ustalenie formatu raportów statusowych
- [ ] Potwierdzenie listy dystrybucyjnej (kto otrzymuje raporty)

### 7. Oczekiwania wobec pierwszego deliverable (10 min)

- [ ] Omówienie pierwszego dostarczenia (co klient zobaczy jako pierwsze)
- [ ] Ustalenie co jest "pierwszym sukcesem" (quick win):
  - [ ] Np. pierwsza faktura wysłana do KSeF sandbox
  - [ ] Np. pierwsza klasyfikacja emaila przez agenta
- [ ] Termin pierwszego deliverable: _______________
- [ ] Forma prezentacji (demo, raport, email)

### 8. Pytania i zakończenie (5-10 min)

- [ ] Otwarta sesja pytań
- [ ] Podsumowanie ustaleń
- [ ] Potwierdzenie najbliższych kroków
- [ ] Podziękowanie za spotkanie

---

## Po spotkaniu

### W dniu spotkania
- [ ] Wysłanie notatek ze spotkania do wszystkich uczestników
- [ ] Aktualizacja CRM (status, notatki)
- [ ] Utworzenie kanału Slack/Teams (jeśli ustalono)
- [ ] Dodanie klienta do kanału komunikacji

### W ciągu 2 dni roboczych
- [ ] Wysłanie formalnego podsumowania ustaleń (email)
- [ ] Przesłanie zaktualizowanego harmonogramu
- [ ] Utworzenie projektu w narzędziu do zarządzania (Jira / Asana / inne)
- [ ] Przydzielenie zadań zespołowi
- [ ] Wysłanie formularza zbierania dostępów do klienta
- [ ] Zaplanowanie pierwszego spotkania statusowego

### W ciągu tygodnia
- [ ] Follow-up dotyczący dostępów (jeśli nie otrzymane)
- [ ] Rozpoczęcie audytu technicznego (jeśli dostępy gotowe)
- [ ] Aktualizacja raportu diagnostycznego (jeśli nowe informacje)
